# iOS 微信 WebView 底部导航栏问题记录

## 问题描述

在 iOS 微信 WebView 中，底部存在一个"白色区域"（非标准安全区），当用户慢速上滑时，这个白色区域会逐渐消失，导致底部导航栏（黑色区域）和白色区域之间出现"分离"和"跳动"现象。

### 核心表现

1. 往上滑动时，白色区域逐渐下降消失，黑色导航栏应该跟随下降
2. 白色区域完全消失后，黑色导航栏应该保持在底部
3. 往下滑动时，白色区域逐渐上浮，黑色导航栏应该被白色区域"顶"上来
4. 整个过程中，黑色导航栏和白色区域之间不应有空隙

### 理想行为原则

1. **黑色导航栏始终显示**：不应出现消失又出现的情况
2. **高度保持不变**：导航栏自身高度不应变化
3. **紧贴白色区域**：但凡有白色区域出现，黑色导航栏底部应始终紧贴白色区域顶部

## 技术分析

### 关键 API

- `window.visualViewport.pageTop`：视觉视口相对于布局视口的偏移量
- `window.innerHeight`：可视区域高度（当白色区域消失时会突然增大约 83px）

### 核心难点

1. **innerHeight 变化滞后**：`window.innerHeight` 的变化不是渐变的，而是突然跳变（671 → 754）
2. **时机不可预测**：无法精确预测 innerHeight 何时变化
3. **往上滑和往下滑逻辑不同**：
   - 往上滑时（innerH = 671）：需要 translateY > 0 跟随白色区域下降
   - 往下滑后（innerH = 671）：需要 translateY = 0 因为白色区域已经完全出现

## 尝试过的方案

### 方案 1：统一公式

```javascript
translateY = min(pageTop * factor, safeArea) - (innerH - initialInnerH);
```

- **问题**：当 innerH 变回 671 时，translateY 会突然跳变

### 方案 2：方向判断

记录滑动方向，区分往上滑和往下滑的逻辑

- **问题**：方向判断和 innerH 变化不同步，仍有跳动

### 方案 3：状态机

定义 4 个状态：initial → scrolling_up → white_gone → scrolling_down

- **问题**：状态转换时 translateY 仍有跳变

### 方案 4：从开始往下滑的位置计算

记录开始往下滑时的 pageTop，从该位置开始计算 translateY

- **问题**：白色区域和黑色区域之间仍有约 20px 的空隙

### 方案 5：简化状态机 + 统一公式

简化为 3 个状态，使用统一公式

- **问题**：跳动和空隙问题仍未完全解决

## 测试数据

基于 iPhone 14 Pro Max 实测：

- 初始 innerHeight：671px
- 白色区域消失后 innerHeight：754px
- 安全区高度 (safeArea)：83px
- 最大 pageTop (maxPT)：约 123px
- 因子 (factor)：约 0.67

## 当前状态

**已放弃特殊处理**，原因：

1. innerHeight 变化时机不可预测，无法实现完美同步
2. 多次尝试均无法完全解决跳动和空隙问题
3. 投入产出比不高

**当前实现**：只保留常规的安全区处理 `paddingBottom: env(safe-area-inset-bottom, 0px)`

## 后续建议

1. 如果微信官方提供更精确的 API，可重新尝试
2. 可考虑使用原生小程序组件替代 H5 实现
3. 或者接受当前的默认行为
