## 六、750px 响应式系统

### 6.1 设计基准

- **设计稿宽度**：750px（微信小程序标准）
- **换算关系**：在 750px 设计稿中，1px ≈ 2rpx
- **Web 端实现**：使用 CSS 变量 `--rpx` 实现响应式适配

### 6.2 核心 CSS 变量

```css
--rpx: calc(100vw / 750); /* 小屏使用 vw */
/* 大屏（>750px）时固定为 1px，防止元素过大 */
```

### 6.3 使用方式

```tsx
// 方式1：使用 rpx 工具函数（推荐）
import { rpx } from '../config/styles';
<div style={{ padding: rpx(48) }}>  {/* 48rpx ≈ 24px */}

// 方式2：直接使用 CSS 变量
<div style={{ padding: 'calc(var(--rpx) * 48)' }}>
```

### 6.4 最大宽度限制

```css
.app-container {
  max-width: 750px;
  margin: 0 auto;
}
```

## 八、底部导航栏与固定底部元素

> 本节为 2026-06 最新结论，作为后续开发（含 AI 协作）的统一参考。

### 8.1 核心原则：能不加就不加

- 底部全局导航（Tab 栏：首页 / 人类手册 / 新人生之路 / 明镜）**只在「模块落地页 / Tab 落点」出现**。
- 进入子页后**一律不再携带**底部导航（类比支付宝：首页有 Tab，点进各子页后普遍没有 Tab，返回靠顶部返回键逐级后退，通常 2–4 步即可回到落地页）。
- 原因：底部导航**占视线、占空间**。它是「成本」，不是「免费的便利」，要克制使用。
- 默认值：`getNavConfig` 兜底为 `showNav: false`。某个子页日后确有「快速回落地页 / 跳其他模块」的强需求时，再单独把该页 `showNav` 打开。

### 8.2 配置方式（每页可配）

- 单一数据源：`src/app/config/navigation-config.ts` 的 `NavConfig`：
  - `showNav`：本页是否显示底部导航。
  - `scrollHide`：是否「下滑隐藏、上滑显示」。
  - `activeIndex`：高亮的 Tab。
- 统一托管：底部导航由 `App.tsx` **中央渲染一次**（当前 scope 限手册馆 `HANDBOOK_NAV_PAGES`），**禁止在页面内各自渲染**，避免重复、不一致与双层底部冲突。

### 8.3 滚动隐藏

- 显示导航的页面统一启用「下滑隐藏、上滑显示」（`scrollHide: true`），保持一致，不要出现「有的页能隐藏、有的不能」。
- 检测用 `src/app/hooks/useScrollDirection.ts`：**捕获阶段**监听 scroll，兼容内层 `overflow:auto` 容器（SPA 常见，window 不滚动）。

### 8.4 高度与安全区

- 导航占位高度 = `calc(env(safe-area-inset-bottom) + 112rpx)`，与 `BottomNavigation` 实际渲染高度一致。
- 改导航高度/内边距时，**必须同步**此常量（`App.tsx` 下发的 `navHeight`），否则会出现「按钮被压住一截」。

### 8.5 固定底部元素「跟随导航」

页面里的固定底部按钮 / 操作坞（如「阅读原文」「继续阅读」「生成阅读建议」）：

- 既**不能被导航遮挡**，也**不能在导航隐藏时留下空隙**。
- 统一消费 `src/app/components/navigation/BottomNavContext.tsx`（`present / hidden / height`）：
  - 导航显示时：上抬到导航**之上**；
  - 导航滚动隐藏 / 本页无导航时：以 `transition: bottom` **柔和落回视口底部**（带安全区内边距）。
- 滚动内容底部需预留 ≥ 固定元素（及导航）高度的 `padding-bottom`，确保最后一屏不被遮挡。

## 九、交互原则：内容被遮挡时自动露出

> 这是核心交互设计原则之一，适用于所有页面，不限于具体某个 case。

**原则**：当一次 UI 变化（展开新区块、插入内容、键盘弹出、新元素出现等）导致新内容被固定底部元素（按钮坞 / 导航 / 键盘）遮挡时，**必须自动滚动让其完整露出**，绝不让用户手动上滑去「找」被挡住的部分。被遮挡会带来「模糊、不清晰、要多操作」的负面体验。

**要点**：

- **判断**：在展开 / 出现动作后，检测目标是否落入被遮挡区域（视口底部被固定元素覆盖的范围）。简化做法：把所在滚动容器柔和滚到底部（前提是容器底部已预留 ≥ 固定元素高度的 `padding-bottom`）。
- **柔和跟随**：滚动要「跟随」而非「跳变」——用 `requestAnimationFrame` 缓动（lerp）持续把容器带到位，时长略长于展开动画，与「一切交互都要轻柔、不突兀」的硬性原则一致。
- **仅在「展开 / 出现」时触发**：收起时不滚动；因数据恢复（hydration）等**非用户主动操作**导致的状态变化也不打扰用户。
- **参考实现**：`src/app/pages/HandbookPractice.tsx` 的 `followToBottom` / `handleToggleJournal`（点「写下自照」展开输入框时，自动滚动使其完整露出，不被底部两枚按钮遮挡）。

## 十三、滚动架构与底部抽屉：根因复盘与定论

> 记录于 2026-06-06。一次耗时数小时、反复返工的教训。这里**区分两类问题，不要混为一谈**：
>
> - **A 类（已解决）**：底部抽屉的「关闭守卫 / 滚动锁」隐患 —— 已定位并修复，见 §13.2–§13.5。
> - **B 类（未确证，暂缓备案）**：桌面端「鼠标按住 / 触控板 Force Click」触发的**全局**滚动
>   卡死 —— 多轮排查仍未确证根因，当前作为**已知问题暂缓处理**，见 §13.7。
>
> 早先版本中「改用 vaul」「优先用成熟库锁滚动」「全局禁用选区即可根治 B 类」等结论**均已被
> 推翻**，以本节最新内容为准。

### 13.1 我们的滚动架构：内层容器滚动（这是对的，不用改）

本应用所有页面采用**内层容器滚动**：页面根 `height:100vh`，内部一个 `div`
用 `overflow:auto` 自己滚动，`body` 本身从不滚动（阅读器即 `scrollRef` 那个 div）。

这是移动端常见且合理的做法，便于固定顶栏/底栏、做沉浸式布局、规避 iOS Safari
地址栏伸缩导致的滚动跳动。**结论：内层容器滚动本身没有错，无需重构为 body 滚动。**

### 13.2 事故现象

反复弹出/关闭底部抽屉后出现两类卡死：

1. 关闭后**第一下滚不动**，再滚一两下才恢复；
2. 开关若干次（第 5/10/20+ 次，随机）后，整页**彻底滚不动、按钮也点不动**；
   刷新可恢复，切页后复发。

### 13.3 抽屉层面的两个真实隐患（A 类，已解决）

> 注意：这两点是「反复开关抽屉后第一下滚不动 / 偶发卡死」的真实成因，**已修**。
> 它们**不是** §13.7「按住/长按全局卡死」的根因（移除后者仍复现）。

**第一层（已修）——自己加的「关闭守卫」**：
为挡长按松手的幽灵点击，早先在阅读器加了「打开后 500ms 内忽略关闭」的时间守卫，
结果把正常关闭也吞掉，`visible` 被钉死 → 抽屉常驻 → 遮罩永久拦截。**已彻底删除。**

**第二层（关键）——成熟弹层库的「滚动锁」与内层容器滚动天生冲突**：
成熟库（vaul / Radix）「防止背景穿透滚动」依赖 `react-remove-scroll`，其设计前提是
**「页面滚 body」**：它会锁 `body` 并挂一批**非 passive 的 `wheel/touchmove` 监听**去
`preventDefault`。

- 这些监听针对的是 body/document，对我们的内层滚动容器是「误伤」；
- 关闭瞬间监听尚未移除 → 吃掉第一个滚动事件（症状一）；
- 快速反复开关时，退出动画未结束又开启，多个 `react-remove-scroll` 实例叠加、监听
  **配平失败而泄漏残留** → `preventDefault` 永久挂着 → 整页彻底卡死（症状二）。

> 教训纠偏：之前误以为「用成熟库就不会有这种问题」。实情是——**成熟库的滚动锁是为
> 「body 滚动」设计的，套在「内层容器滚动」架构上反而制造问题**。库没错，但它的前提
> 假设和我们的架构不匹配。

### 13.4 最终定论：底部抽屉「不需要任何滚动锁」

关键认知：**一个铺满全屏的固定遮罩，天然就挡住了背景滚动**——遮罩的滚动祖先是
`body`，而非内层滚动容器，手指/滚轮落在遮罩上根本带不动背景。所以底部抽屉**根本
不需要 body 锁 / `react-remove-scroll` / 任何全局 `wheel/touchmove` 监听**。

`BottomSheet` 的最终实现（`components/shared/BottomSheet.tsx`，自研但极简）：

- **Portal 到 body**；遮罩（半透明、点击关闭）+ 面板（`transform` 升降）；
- 面板 `overflow-y:auto` + `overscroll-behavior:contain`，内部滚动不串联背景；
- **零 body 锁、零全局监听、零 `react-remove-scroll`**；
- 遮罩/容器的 `pointer-events` **直接绑定父级 `visible`**：关闭即放行，即便万一未及时
  卸载也绝不可能拦截背景；动画结束后再卸载。
- 关闭后**没有任何残留**（无监听、无 body 样式、无锁计数），两类卡死从机制上不可能再现。

### 13.5 规约（务必遵守）

- **底部抽屉/弹层统一用 `components/shared/BottomSheet`**，不要在内层容器滚动架构里
  引入会锁 body 的弹层库（vaul / Radix Dialog 默认 modal / `react-remove-scroll` 等）。
- **永远不要加任何「拦截关闭」的逻辑**（时间窗守卫、条件 return 等）：关闭必须即时生效，
  绝不能让 `visible` 被钉死。
- 任何浮层/遮罩：**`pointer-events` 必须绑定「可见态」（淡入完成的 `shown`），而非仅绑 `visible`**。
  否则会出现「`visible=true` 但动画态 `shown=false`」的**隐形遮罩**——肉眼看不见却仍拦截，
  表现为「看不到弹框却整页滚不动」。**铁律：看不见就绝不能拦截。**
- 自身逻辑必须自包含，**消失后不得对背景页/上层造成任何残留影响**（监听、锁、被钉死的状态等）。
- 「优先用成熟库」仍是好原则，但**前提是该库的假设与本项目架构匹配**；不匹配时，一个
  职责单一、无副作用的极简自研组件，反而更简单、更可控、更稳。

### 13.6 产品决策：移除「长按段落」交互，笔记收敛到章节级（2026-06-06）

「长按段落 → 升起注解条」这一交互本身就不稳定（长按有时升起又自己回落），且在移动端长按、
原生选区、合成 click、滚动判定之间耦合很深、价值有限。**经评估其必要性后整体移除**——
读者真正需要的是「分享 / 记录」，而这在**章节级**就能更干净地满足，无需逐段交互。

> 说明：移除它**并不能**解决 §13.7 的「按住/长按全局卡死」（移除后仍复现），所以这是一次
> 独立的**产品取舍**，不是 bug 修复。复盘后维持「移除，不再还原」。

- 阅读器正文回归最纯粹的「可滚动文本」：段落无任何指针手势 / highlight 态。
- 段落级「划线 / 想法」整套删除（含 `useHandbookNotes`、`ParagraphAnnotateSheet`）。

**笔记 vs 自照（相互独立，不可混用）：**

- **笔记**：针对【整章内容】的自由记录。底栏「笔记」入口打开 `ChapterNoteSheet`，
  进入即显示已有笔记可续写；点「保存」或**有改动时收起抽屉**都会写入（`useChapterNote`，
  localStorage，每章一条覆盖式，字数上限 `CHAPTER_NOTE_MAX=1000`）。
  底栏「笔记」图标在该章**已记**时显示金色圆点 + 转金色，便于读了很多章后一眼区分。
- **自照**：练习里针对【练习本身】的自我观照，归属练习模块，与笔记互不相干。

**分享：**

- 卷级分享在卷页右上角（既有）。章级分享新增于**阅读器顶栏右上角**（`Share2`），
  复用 `ShareQuoteSheet` 生成卡片，内容取该章导言（章节级，而非某一段），落款
  `《卷名》· 章名`。「金句卡」本质是段落级摘录卡，已随长按交互一并退场。

### 13.7 已知问题（备案）：桌面端「按住 / 长按」触发的全局滚动卡死 —— 根因未确证，暂缓

> 状态：**未解决、暂缓处理**。本平台移动优先，该问题目前只在桌面端非主路径上复现；
> 在确证根因前**不要再做投机式修补**。本节为备案，便于日后（或他人）接手时不重蹈覆辙。

**现象**：在桌面浏览器（已知至少 macOS）里，鼠标在页面上（**空白处或文字处都会**）**按住**、
或触控板 **Force Click**，松开后整页**滚不动**；且此后**切到任意其它页也滚不动，刷新才恢复**。
手册首页、卷页（章节列表）、阅读器都能复现。按住时可能弹出系统菜单（Look Up / 复制 / 翻译…）。

**判断**：症状「跨页持续、刷新才恢复」说明它**不是某个页面/组件的状态**，而是**浏览器 document
层的输入态**。这也解释了为何逐一移除组件逻辑都无效——它们都不是根因：

- 移除「长按段落」整套交互（§13.6）→ 仍复现；
- 把阅读器改成「纯滚动页 / 顶底栏不翻转」→ 仍复现（该改动已**还原**为原沉浸式 chrome）；
- 外壳层全局 `user-select:none` + `touch-callout:none` → **仍复现**，且副作用是**正文无法
  选中复制**，已**还原**；
- 诊断探针（已移除）显示故障时无遮罩残留、`body/html` 样式正常、无监听泄漏。

**当前最佳假设（仍未确证，根因属"其它未知原因"）**：

- macOS 桌面端 + 内层容器滚动架构下，「按住 / Force Click」可能让浏览器进入一种**未正常结束的
  指针/选区拖拽态**，在 document 层吞掉后续滚动。——但「空白处按下并未选中任何文字」与纯
  选区解释**相矛盾**，故**不能确认**。除此之外暂无更有力的解释，根因仍**待查**。

> 注意：此卡死与 PWA / Service Worker **无关**——这是两件事，**不要混为一谈**。SW 缓存只是一个
> 与本问题无关的、尚未发生的潜在风险，已单独记于 §13.8 备查。

**决策（暂缓 + 待验证项）**：

- 该卡死目前只在**桌面鼠标按住 / 触控板 Force Click**这种非移动端主路径复现。推断移动端
  触摸滚动 + 长按（手指抬起即结束一次触摸序列）**大概率不残留此态**，但**尚未经真机 / WebView
  验证**——这是暂缓的前提，必须后续在真机确认。
- 待办：① 真机验证（iOS Safari / Android Chrome / App WebView）移动端是否复现；② 若移动端复现
  或桌面需根治，再专项排查 `webkitmouseforcedown` / `contextmenu` / `selectstart` 事件链与
  内层滚动的相互作用。

> 教训：当一个症状**跨页持续、刷新才恢复**时，根因几乎必然在**浏览器/系统输入态**，而非页面
> 组件——此时逐页/逐组件打补丁都是徒劳。应先在真机上确认影响范围，再决定「根治」还是「按
> 平台主路径暂缓」，避免无限投入与投机式全局改动（如本例的全局 `user-select:none`，既没解决
> 问题又牺牲了正文选区）。

### 13.8 潜在事项（预案）：PWA Service Worker 缓存（**目前未发现实际问题**）

> 状态：**当前未观察到任何相关问题**，本节仅为**预案**——若日后线上出现「明明已更新部署、
> 但用户仍看到旧版本 / 改动疑似没生效」的现象，可回到这里参考。**它与 §13.7 的滚动卡死无关。**

本前端是 H5（部署在阿里云，经微信内置浏览器 / Safari 等打开），并启用了 `vite-plugin-pwa`。
Service Worker 会缓存静态资源以支持离线/秒开，理论上**可能**带来一个常见副作用：

- 新版本部署后，用户设备上的旧 SW 仍可能先用**已缓存的旧 JS/CSS** 响应，需等 SW 更新并在
  下次（或再下次）打开时才换上新版 → 表现为「更新没生效」。微信 WebView 还可能叠加自己的
  资源缓存，更易让人误判。

**排查/缓解（真出现时再处理，避免过度设计）**：

- 测试核对：用**无痕窗口**或清掉站点数据后复测，确认看到的是不是旧产物。
- 配置项：在 `vite-plugin-pwa` 用 `registerType: 'autoUpdate'`，并视情况开启
  `workbox.skipWaiting` + `clientsClaim`，让新 SW 尽快接管；必要时给页面加「发现新版本→提示
  刷新」。
- 部署核对：确认构建产物带内容哈希、HTML 不被强缓存（`Cache-Control` 合理），避免 CDN/服务器
  层把入口 HTML 也长缓存。

> 再次强调：以上**均为预防性记录**，目前并无证据表明已发生；**不要**把它当作任何已知 bug 的
> 成因（尤其不要与 §13.7 滚动卡死挂钩）。

## 十四、结缘分享：当前形态与上线后形态（2026-06-06）

**定位**：分享是「结缘」（让文字安静流动），不是流量/社交表演。

**当前形态（数据层未就绪）**：`ShareQuoteSheet` 生成一张安静的卡片——
导言/金句 + 出处 + 署名 + **二维码占位 + 「扫码读全文」链接位**；提供「复制文字 / 保存图片」。
卷级分享在卷页右上角，章级分享在阅读器顶栏右上角。UI **已按最终形态呈现**，仅链接/二维码
为占位（`shareUrl` 缺省 → 显示「完整章节链接即将开放」）。

**上线后形态（待后端）**：

1. 分享时后端为该章生成一个**免登录公开只读短链**（如 `/r/{token}`）。收件人点开/扫码即可
   在网页**完整、免权限**读完整章正文；页面底部再引导「打开 App / 了解更多」。
2. 前端把该短链回填到 `ShareQuoteSheet` 的 `shareUrl`：
   - 预览卡 / 保存的图片：用真实二维码替换占位（接入轻量 QR 生成）；
   - 复制文字：附带「阅读整章全文 ▸ {shareUrl}」。
3. 图片卡只承担社媒「封面/钩子」，**真正内容承载在链接**，从而解决「只发一句导言对方看不到正文」。
4. 「全文只读」vs「前几段预览 + 打开 App 看全文」是运营取舍；按平台「结缘而非流量」气质，
   倾向**全文只读**以最大化自然传播。

> 预留点：`ShareQuoteSheet` 已开放 `shareUrl` 入参与二维码槽位；后端就绪后只需回填链接 +
> 接入真实 QR，无需改动交互结构。
