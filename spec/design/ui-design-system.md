## 六、750px 响应式系统

### 6.1 设计基准

- **设计稿宽度**：750px（微信小程序标准）
- **换算关系**：在 750px 设计稿中，1px ≈ 2rpx
- **Web 端实现**：使用 CSS 变量 `--rpx` 实现响应式适配

### 6.2 核心 CSS 变量

```css
--rpx: calc(100vw / 750); /* 小屏使用 vw */
/* 大屏（>750px）时固定为 1px，防止元素过大 */
```

### 6.3 使用方式

```tsx
// 方式1：使用 rpx 工具函数（推荐）
import { rpx } from '../config/styles';
<div style={{ padding: rpx(48) }}>  {/* 48rpx ≈ 24px */}

// 方式2：直接使用 CSS 变量
<div style={{ padding: 'calc(var(--rpx) * 48)' }}>
```

### 6.4 最大宽度限制

```css
.app-container {
  max-width: 750px;
  margin: 0 auto;
}
```

## 八、底部导航栏与固定底部元素

> 本节为 2026-06 最新结论，作为后续开发（含 AI 协作）的统一参考。

### 8.1 核心原则：能不加就不加

- 底部全局导航（Tab 栏：首页 / 人类手册 / 新人生之路 / 明镜）**只在「模块落地页 / Tab 落点」出现**。
- 进入子页后**一律不再携带**底部导航（类比支付宝：首页有 Tab，点进各子页后普遍没有 Tab，返回靠顶部返回键逐级后退，通常 2–4 步即可回到落地页）。
- 原因：底部导航**占视线、占空间**。它是「成本」，不是「免费的便利」，要克制使用。
- 默认值：`getNavConfig` 兜底为 `showNav: false`。某个子页日后确有「快速回落地页 / 跳其他模块」的强需求时，再单独把该页 `showNav` 打开。

### 8.2 配置方式（每页可配）

- 单一数据源：`src/app/config/navigation-config.ts` 的 `NavConfig`：
  - `showNav`：本页是否显示底部导航。
  - `scrollHide`：是否「下滑隐藏、上滑显示」。
  - `activeIndex`：高亮的 Tab。
- 统一托管：底部导航由 `App.tsx` **中央渲染一次**（当前 scope 限手册馆 `HANDBOOK_NAV_PAGES`），**禁止在页面内各自渲染**，避免重复、不一致与双层底部冲突。

### 8.3 滚动隐藏

- 显示导航的页面统一启用「下滑隐藏、上滑显示」（`scrollHide: true`），保持一致，不要出现「有的页能隐藏、有的不能」。
- 检测用 `src/app/hooks/useScrollDirection.ts`：**捕获阶段**监听 scroll，兼容内层 `overflow:auto` 容器（SPA 常见，window 不滚动）。

### 8.4 高度与安全区

- 导航占位高度 = `calc(env(safe-area-inset-bottom) + 112rpx)`，与 `BottomNavigation` 实际渲染高度一致。
- 改导航高度/内边距时，**必须同步**此常量（`App.tsx` 下发的 `navHeight`），否则会出现「按钮被压住一截」。

### 8.5 固定底部元素「跟随导航」

页面里的固定底部按钮 / 操作坞（如「阅读原文」「继续阅读」「生成阅读建议」）：

- 既**不能被导航遮挡**，也**不能在导航隐藏时留下空隙**。
- 统一消费 `src/app/components/navigation/BottomNavContext.tsx`（`present / hidden / height`）：
  - 导航显示时：上抬到导航**之上**；
  - 导航滚动隐藏 / 本页无导航时：以 `transition: bottom` **柔和落回视口底部**（带安全区内边距）。
- 滚动内容底部需预留 ≥ 固定元素（及导航）高度的 `padding-bottom`，确保最后一屏不被遮挡。

## 九、交互原则：内容被遮挡时自动露出

> 这是核心交互设计原则之一，适用于所有页面，不限于具体某个 case。

**原则**：当一次 UI 变化（展开新区块、插入内容、键盘弹出、新元素出现等）导致新内容被固定底部元素（按钮坞 / 导航 / 键盘）遮挡时，**必须自动滚动让其完整露出**，绝不让用户手动上滑去「找」被挡住的部分。被遮挡会带来「模糊、不清晰、要多操作」的负面体验。

**要点**：

- **判断**：在展开 / 出现动作后，检测目标是否落入被遮挡区域（视口底部被固定元素覆盖的范围）。简化做法：把所在滚动容器柔和滚到底部（前提是容器底部已预留 ≥ 固定元素高度的 `padding-bottom`）。
- **柔和跟随**：滚动要「跟随」而非「跳变」——用 `requestAnimationFrame` 缓动（lerp）持续把容器带到位，时长略长于展开动画，与「一切交互都要轻柔、不突兀」的硬性原则一致。
- **仅在「展开 / 出现」时触发**：收起时不滚动；因数据恢复（hydration）等**非用户主动操作**导致的状态变化也不打扰用户。
- **参考实现**：`src/app/pages/HandbookPractice.tsx` 的 `followToBottom` / `handleToggleJournal`（点「写下自照」展开输入框时，自动滚动使其完整露出，不被底部两枚按钮遮挡）。

## 十、阅读方向策略

| 元素类型   | 方向   | 说明                       |
| ---------- | ------ | -------------------------- |
| 功能性元素 | 左→右  | 卡片入口、导航栏、按钮     |
| 意境性元素 | 右→左  | 节气竖排、装饰文字         |
| 循环性元素 | 无方向 | 呼吸圆环，通过动画节奏引导 |

**三卡片顺序**：徐→止→定（左→右），由动入静的渐修路径。

## 十二、设计检查清单

### 新建组件时

- [ ] 使用 `config/styles.ts` 中的共享常量
- [ ] 文本/图片放在 config 层而非硬编码
- [ ] 圆角使用 CSS 变量（`--radius-*`）
- [ ] 用阴影而非边框
- [ ] 注释清晰（用途、Props、使用场景）

### 新建页面时

- [ ] 基于 750px 设计基准
- [ ] 页面只负责组装，不含硬编码数据
- [ ] 宋体字只用于标题/金句
- [ ] 底部导航：默认不加，仅落地页/确有需要的页才 `showNav: true`（见 §八）
- [ ] 固定底部按钮/坞通过 `BottomNavContext` 跟随导航升降，不遮挡内容（见 §8.5）
- [ ] 任何 UI 变化后若内容被底部固定元素遮挡，自动柔和滚动露出（见 §九）
